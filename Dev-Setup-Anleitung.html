Developer-Setup – Strategy Builder V2

Lokale Vollumgebung mit BSC-Fork · frisch nach git clone · Stand: 15.06.2026

Von null bis zur laufenden Full-Stack-Umgebung (Contracts-Fork + Backend + Frontend + DB). Die Reihenfolge ist wichtig: erst Fork & Deploy, dann Adressen in die .env-Dateien, dann Services starten — sonst laufen Backend/Frontend mit leeren Contract-Adressen.

Goldene Regel: Fork → Deploy → .env füllen → Seed → Services. Jeder Schritt unten baut darauf auf.

1 · Dependencies installieren

pnpm install

Installiert alle Workspace-Pakete (packages/*) und baut die nativen Deps (Prisma, esbuild, keytar, NestJS). Generiert den Prisma-Client.

2 · Env-Dateien aus den Vorlagen anlegen

Für jedes Paket existiert eine .env.example. Die Contract-Adressen bleiben vorerst leer und werden nach dem Deploy (Schritt 4) eingetragen.

cp packages/contracts/.env.example packages/contracts/.env
cp packages/backend/.env.example  packages/backend/.env
cp packages/frontend/.env.example packages/frontend/.env
Jetzt das Wichtigste in packages/contracts/.env setzen: einen archivfähigen BSC-RPC. Öffentliche RPCs (bsc-dataseed.binance.org) scheitern beim Forken mit „missing trie node". 
# packages/contracts/.env
BSC_MAINNET_RPC_URL=https://bsc-mainnet.public.blastapi.io   # oder eigener Alchemy/QuickNode-Key
DEPLOYER_PRIVATE_KEY=0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80  # Hardhat #0, nur lokal!

3 · BSC-Fork starten Terminal 1

pnpm contracts:fork:bsc

Startet einen Hardhat-Node, geforkt von BSC-Mainnet, auf http://localhost:8545 (Chain-ID 31337). Terminal offen lassen.

4 · Contracts deployen Terminal 2

Sobald der Fork läuft, das komplette System deployen (kompiliert automatisch mit production build profile — ohne Optimizer schlägt das Proxy-Deployment fehl):

pnpm contracts:deploy:fork

Deployt FeeRegistry, Vault-Factory + Implementation, MockPriceOracle, die Beispiel-Conditions/Actions und die 2 DeFi-Registries + 11 DeFi-Actions (Aave V3 + PancakeSwap V3 inkl. Swap-to-Range-Ratio & Wick-&-Wait-Condition), konfiguriert Fees + Gas-Kompensation und fundet das Test-Wallet. Alle Adressen werden ausgegeben und nach packages/contracts/deployments/fork-latest.json geschrieben.

Nur DeFi-Actions nachziehen (Basissystem steht schon)? — npx hardhat run --network localhost scripts/deploy-defi-actions.ts deployt nur die Registries + Actions und merged die Adressen in fork-latest.json (Factory/Vaults bleiben unberührt). Danach neu seeden.

5 · Adressen in die Service-Env-Dateien übertragen

Aus der Deploy-Ausgabe (bzw. fork-latest.json) die Adressen kopieren:

packages/backend/.env
NODE_ENV=development            # Pflicht: Balance-Reads über lokales RPC
DATABASE_URL=postgresql://pecunity:pecunity@localhost:5432/pecunity
RPC_URL=http://localhost:8545
FACTORY_ADDRESS=0x...           # StrategyBuilderVaultFactory
FEE_REGISTRY_ADDRESS=0x...      # FeeRegistry
INDEXER_CONFIRMATIONS=0         # Pflicht am Fork (idle fork mined keine Confirmations)
JWT_SECRET=<zufälliger-hex-wert>
FRONTEND_URL=http://localhost:5173
PORT=3001
packages/frontend/.env
VITE_API_URL=http://localhost:3001
VITE_FACTORY_ADDRESS=0x...      # gleiche Factory-Adresse
INDEXER_CONFIRMATIONS=0 nicht vergessen: Sonst erscheinen am Fork frische Deposits/Executions nie in der History (ein idle Fork mined die Confirmation-Blöcke nicht).

6 · Datenbank, Migration & Services starten Terminal 3

pnpm dev

Ein Befehl erledigt: Backend-.env aus Vorlage anlegen (falls fehlt) · PostgreSQL via Docker hochfahren · auf DB warten · prisma migrate dev · shared bauen · Backend (:3001) + Frontend (:5173) im Watch-Mode starten.

7 · StepType-Katalog seeden einmalig, nach dem Deploy

Der Seed liest die Adressen aus fork-latest.json — deshalb nach Deploy & mit laufender DB ausführen (z. B. in einem 4. Terminal, während pnpm dev läuft):

pnpm db:seed

Seedet eine Zeile pro Step (3 Conditions + Beispiel-Actions + DeFi-Actions) + die Token-Listen und überspringt alles, was noch auf address(0) steht. Ohne Seed nach einem Deploy encoden neue Automations die toten Alt-Adressen und reverten.

Nach jedem erneuten Full-Deploy neu seeden. Bei einem Redeploy bekommen Contracts neue Adressen; alte StepType-Zeilen können sonst als Duplikate im Editor hängenbleiben. Tipp: für bestehende Setups lieber inkrementell deploy-defi-actions.ts nutzen.

8 · MetaMask verbinden

Eigenes Netzwerk hinzufügen: RPC http://localhost:8545 · Chain-ID 31337 · Currency ETH. Das Frontend schaltet im Dev-Modus automatisch auf Chain 31337.

Test-Account importieren — das Deploy-Skript fundet 0xBcd4042DE499D14e55001CcbB24a551F3b954096 mit 150 USDT + 1 WBNB:

# Hardhat Account #5
0x47e179ec197488593b187f80a00eb0da91f1b9d0b13f8733639f19c30a34926a

Dann http://localhost:5173 öffnen und Wallet verbinden.

Fertig. Du hast jetzt: Fork-Node (T1), Deploy erledigt (T2), DB+Backend+Frontend (T3), Katalog geseedet. Anmelden via SIWE, Vault anlegen, Automations bauen.

9 · Optional – Keeper für Public-Automations

Public-Automations (mit Trigger) werden von einem externen Keeper ausgeführt, sobald der Trigger erfüllt ist. Am Fork manuell anstoßen:

pnpm contracts:execute:fork

Synct zuerst die Fork-Uhr auf Wall-Clock, führt dann jede aktive, public, getriggerte Automation aus und meldet Fehlschläge (Reverts) ans Backend (wenn KEEPER_INGEST_SECRET in beiden .env gesetzt ist). Voraussetzung: die Vault hat eine Gas-Reserve (minFeeDeposit > 0 + gefüllt), sonst Revert InsufficientFeeDeposit.

14 · Troubleshooting

SymptomUrsache / Lösung
Fork-Start: „missing trie node"Kein Archiv-RPC. BSC_MAINNET_RPC_URL auf BlastAPI/Alchemy/QuickNode setzen.
Deploy bricht mit StackOverflow abproduction build profile nötig — contracts:deploy:fork macht das automatisch; nicht den nackten hardhat run ohne --build-profile production nutzen.
Editor zeigt nur manche / doppelte Step-NodesDeFi-Actions nicht deployed bzw. Alt-Adressen in der DB. Deployen + pnpm db:seed (bei Duplikaten Tabelle vorher leeren).
Neue Automation revertet sofort / Keeper „trigger not met"StepTypes nach dem Deploy nicht neu geseedet → tote Alt-Adressen. Neu seeden und Automation neu deployen.
Deposits/Executions tauchen nie in der History aufINDEXER_CONFIRMATIONS=0 in der Backend-.env setzen, Backend neu starten.
Portfolio/Balances bleiben leerNODE_ENV=development in der Backend-.env (liest dann über lokales RPC statt Alchemy).
Geänderte .env wirkt nichtpnpm dev neu starten (Env wird beim Start gelesen).
Docker-Fehler bei pnpm devDocker-Daemon starten (docker ps muss laufen).
